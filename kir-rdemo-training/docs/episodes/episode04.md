# Episode 4: Ship It

## Learning Objectives

By the end of this episode you will be able to:

- Build a `pkgdown` documentation website from your existing roxygen2 docs
- Set up GitHub Actions to run `R CMD check` on every push
- Deploy your `pkgdown` site automatically to GitHub Pages
- Explain how colleagues can install your package with `pak`

---

## 4.1 What Does "Shipping" Mean for an HPC Package?

You're unlikely to submit `kirgcdemo` to CRAN — and you don't need to. For an internal BMRC package, "shipping" means:

- **A URL colleagues can point to** for documentation
- **A one-liner install command** that works from any login node
- **CI that catches regressions** before they hit users

`pkgdown` + GitHub Actions covers all three.

---

## 4.2 Setting Up pkgdown

```r
usethis::use_pkgdown()
```

This creates `_pkgdown.yml` (minimal by default) and adds relevant entries to `.Rbuildignore`.

Build the site locally first:

```r
pkgdown::build_site()
```

Then open `docs/index.html` — you'll see a complete website with:

- A home page built from your `README.md`
- A **Reference** section with every exported function, built from your `man/` pages
- An **Articles** section containing your vignette
- A top-level nav with search

!!! note "Where does the content come from?"
    pkgdown doesn't require you to write anything new. It reads `DESCRIPTION`, `README.md`, `man/*.Rd`, and `vignettes/*.Rmd` — all files you already have. Documentation you write once appears in three places: `?gc_content` in the console, R CMD check examples, and the website.

---

## 4.3 Customising `_pkgdown.yml`

The default site is functional but plain. Add a minimal configuration:

```yaml
# _pkgdown.yml
url: https://kir-rescomp.github.io/kirgcdemo/

template:
  bootstrap: 5

home:
  title: kirgcdemo — DNA Sequence Analysis Utilities

navbar:
  structure:
    left: [intro, reference, articles]
    right: [github]

reference:
  - title: "Sequence Analysis"
    desc: "Core functions for working with DNA sequences"
    contents:
      - gc_content
      - reverse_complement
```

Rebuild after changes:

```r
pkgdown::build_site()
```

## 4.4 GitHub Actions — R CMD check

Push your package to GitHub first:

```r
usethis::use_github()   # creates repo and pushes — requires gh CLI or PAT
```

Then add the standard R-CMD-check workflow:

```r
usethis::use_github_action("check-standard")
```

This creates `.github/workflows/R-CMD-check.yaml`:

```yaml
# .github/workflows/R-CMD-check.yaml  (generated by usethis)
on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]

name: R-CMD-check

jobs:
  R-CMD-check:
    runs-on: ${{ matrix.config.os }}
    strategy:
      matrix:
        config:
          - {os: ubuntu-latest, r: 'release'}
          - {os: ubuntu-latest, r: 'devel'}
          - {os: windows-latest, r: 'release'}
          - {os: macos-latest,   r: 'release'}

    steps:
      - uses: actions/checkout@v4
      - uses: r-lib/actions/setup-r@v2
        with:
          r-version: ${{ matrix.config.r }}
      - uses: r-lib/actions/setup-r-dependencies@v2
        with:
          extra-packages: any::rcmdcheck
      - uses: r-lib/actions/check-r-package@v2
```

On every push, GitHub runs your full `R CMD check` across Linux, Windows, and macOS, on the current release and development versions of R.

---

## 4.5 GitHub Actions — pkgdown Deploy

Add automatic site deployment:

```r
usethis::use_github_action("pkgdown")
```

This creates `.github/workflows/pkgdown.yaml`. After you enable GitHub Pages in your repo settings (source: `gh-pages` branch), every push to `main` will rebuild and redeploy the site.

Your documentation URL will be:
```
https://kir-rescomp.github.io/kirgcdemo/
```

!!! tip "Adding a badge to README"
    ```r
    usethis::use_github_actions_badge("R-CMD-check")
    ```
    This inserts a live status badge into your `README.md` — it turns green when checks pass.

## 4.6 Installing Your Package

Once the package is on GitHub, colleagues install it with `pak`:

```r
# Fast, parallel, handles dependencies correctly
pak::pak("kir-rescomp/kirgcdemo")
```

Or with `remotes` (the traditional approach):

```r
remotes::install_github("kir-rescomp/kirgcdemo")
```

On BMRC, where nodes may not have outbound GitHub access, build a source tarball and distribute it:

```r
devtools::build()   # creates kirgcdemo_0.1.0.tar.gz
```

Then on the cluster:

```bash
module load R/4.5.1-foss-2023a
R CMD INSTALL kirgcdemo_0.1.0.tar.gz
```

Or from within R:

```r
install.packages("kirgcdemo_0.1.0.tar.gz", repos = NULL, type = "source")
```

## 4.7 Versioning

Follow [semantic versioning](https://semver.org/) in `DESCRIPTION`:

```
0.1.0 → initial release
0.1.1 → bug fix
0.2.0 → new function added
1.0.0 → stable API
```

Bump versions with:

```r
usethis::use_version("minor")   # 0.1.0 → 0.2.0
```

Tag releases in git:

```bash
git tag v0.1.0
git push origin v0.1.0
```

In four episodes you've gone from a bioinformatics script to a complete R package:

| Milestone | Tool |
|-----------|------|
| Reproducible environment | `renv` |
